# Beluga Master Plan

## 1. 문서 목적

이 문서는 **선착순 기프티콘 서비스(벨루가)** 프로젝트의 마스터 플랜이다.  
단순 기능 목록이 아니라, 아래 흐름이 한 번에 보이도록 정리한다.

- 어떤 문제를 해결하려는지
- 어떤 범위까지 구현할지
- 정합성과 성능을 어떻게 검증할지
- 어떤 기준으로 개선안을 비교할지
- 포트폴리오 관점에서 무엇을 남길지

---

## 2. 프로젝트 개요

### 2.1 프로젝트명

**선착순 기프티콘 서비스(벨루가)**

### 2.2 한 줄 소개

이벤트 시작 시점에 몰리는 동시 요청 환경에서, **정확한 선착순 당첨 처리와 데이터 정합성**을 보장하기 위해 DB, 트랜잭션, 락 전략을 실험하고 개선하는 백엔드 프로젝트

### 2.3 프로젝트 목적

이 프로젝트의 목적은 특정 이벤트 시작 시점에 많은 사용자가 동시에 접속하는 상황에서, 제한된 인원만 **정확하게 선착순으로 당첨 처리**할 수 있는 시스템을 구현하고 검증하는 것이다.

핵심은 단순 CRUD 구현이 아니라 다음 문제를 직접 재현하고 개선하는 데 있다.

- 동시 요청 환경에서의 정확한 선착순 처리
- 동일 사용자 중복 참여 방지
- 최대 당첨 인원 초과 방지
- 트래픽 집중 시 응답 지연과 DB 병목 분석
- 쿼리, 인덱스, 트랜잭션, 락 전략 개선
- 개선 전후 부하 테스트 기반 성능 비교

### 2.4 프로젝트 성격

- **주제 중심**: 선착순 처리 정합성과 대량 동시성
- **실험 중심**: 구현 후 부하 테스트로 병목을 확인하고 개선
- **측정 중심**: 평균 응답 시간, p95/p99, TPS, 오류율, 정합성 오류 여부를 수치로 비교
- **포트폴리오 중심**: 기술 선택 이유와 개선 근거를 설명 가능한 형태로 문서화

## 3. 해결할 핵심 문제

### 3.1 동시성 정합성 문제

이벤트 시작 시점에 다수의 요청이 거의 동시에 들어오면 다음 문제가 발생할 수 있다.

- 최대 당첨 가능 수량보다 더 많은 사용자가 당첨되는 문제
- 같은 사용자가 여러 번 참여 처리되는 문제
- 실제 처리 순서와 다른 결과가 저장되는 문제
- 재고 확인과 저장이 분리되어 정합성이 깨지는 문제

### 3.2 성능 병목 문제

짧은 시간 동안 요청이 집중되면 다음 병목이 발생할 수 있다.

- 특정 테이블에 대한 쓰기 경쟁 증가
- 중복 체크 및 당첨 처리 과정에서의 쿼리 지연
- 락 경합 증가
- 트랜잭션 처리 시간 증가
- DB 커넥션 부족 또는 응답 지연

### 3.3 핵심 질문

이 프로젝트는 아래 질문에 답할 수 있어야 한다.

- 왜 단순 구현만으로는 선착순 정합성이 깨질 수 있는가?
- 어떤 상황에서 초과 당첨이 발생하는가?
- 중복 참여는 어떻게 차단했는가?
- 어떤 쿼리와 처리 흐름이 병목이었는가?
- 어떤 락 전략을 검토했고 왜 그 방식을 선택했는가?
- 개선 전후 성능 차이는 얼마였는가?
- 아직 남아 있는 한계는 무엇인가?

## 4. 목표와 성공 기준

### 4.1 최상위 목표

1. **정합성 확보**: 어떤 동시 요청 상황에서도 최대 당첨 수와 중복 참여 제약을 지킨다.
2. **성능 확보**: 이벤트 시작 직후의 집중 요청을 측정 가능하고 비교 가능한 형태로 처리한다.
3. **실험 가능 구조 확보**: 쿼리, 인덱스, 락 전략, 트랜잭션 범위를 바꿔가며 반복 검증할 수 있게 만든다.
4. **설명 가능한 결과 확보**: 개선 이유와 결과를 포트폴리오 문서만으로도 이해할 수 있게 남긴다.

### 4.2 성공 기준

- 최대 당첨 인원을 초과하지 않는다.
- 동일 사용자의 중복 참여가 발생하지 않는다.
- 사용자 응답 결과와 DB 저장 결과가 일치한다.
- 동시 요청 상황에서도 결과가 일관되게 저장된다.
- 동일 부하 시나리오로 개선 전후 결과를 비교할 수 있다.
- 기술 선택 이유와 한계를 설명할 수 있다.

- 이벤트 시작 이후 참여 요청을 보낼 수 있다.
- 자신의 참여 결과를 조회할 수 있다.

- 이벤트 참여 및 결과 조회

- 병목 분석, 개선 실험, 결과 비교 문서화

#### 초기 제외 범위

- 복잡한 관리자 UI 완성도
- 실시간 알림, 쿠폰 발송 자동화

### 5.3 초기 시스템 가정

- 단일 애플리케이션 서버
- 단일 RDB 인스턴스

- 부하 테스트 도구 별도 운용
- AWS 배포는 후속 단계에서 적용

## 6. 기능 요구사항

### 6.1 인증 및 회원 관리

- `FR-01` 사용자는 이메일, 비밀번호, 닉네임으로 회원가입할 수 있어야 한다.
- `FR-02` 동일 이메일로 중복 회원가입할 수 없어야 한다.
- `FR-03` 사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다.
- `FR-04` 로그인한 사용자는 마이페이지에서 비밀번호를 수정할 수 있어야 한다.
- `FR-05` 로그인한 사용자는 마이페이지에서 닉네임을 수정할 수 있어야 한다.

### 6.2 사용자 이벤트 조회

- `FR-06` 메인 페이지는 현재 진행 중인 이벤트를 최상단에 우선 노출해야 한다.
- `FR-07` 메인 페이지는 예정된 이벤트를 현재 진행 중인 이벤트 하단에 나열해야 한다.
- `FR-08` 이벤트 목록 항목은 이벤트명, 제공 수량, 시작 일시, 종료 일시 정보를 포함해야 한다.
- `FR-09` 사용자는 별도 이벤트 상세 화면 없이 메인 페이지의 이벤트 목록에서 참여에 필요한 정보를 확인할 수 있어야 한다.

### 6.3 이벤트 참여 및 결과 조회

- `FR-10` 이벤트 시작 전 참여 요청은 실패 처리되어야 한다.
- `FR-11` 이벤트 종료 후 참여 요청은 실패 처리되어야 한다.
- `FR-12` 동일 사용자는 동일 이벤트에 한 번만 참여할 수 있어야 한다.
- `FR-13` 최대 당첨 인원을 초과해 당첨자를 선정하면 안 된다.
- `FR-14` 참여 요청은 선착순 기준에 따라 처리되어야 한다.
- `FR-15` 참여 결과는 당첨, 미당첨, 중복 참여, 이벤트 시작 전, 이벤트 종료, 잘못된 요청, 시스템 처리 실패 중 하나로 반환해야 한다.
- `FR-16` 사용자는 마이페이지에서 자신이 참여한 이벤트 목록과 각 이벤트의 당첨 결과를 조회할 수 있어야 한다.
- `FR-17` 당첨 사용자는 서비스 내에서 기프티콘 이미지를 조회할 수 있어야 한다.
- `FR-18` 당첨 사용자는 기프티콘 이미지를 다운로드할 수 있어야 한다.

### 6.4 관리자 이벤트 운영 및 결과 조회

- `FR-19` 관리자는 대표 이미지, 이벤트명, 상품명, 당첨 인원 수, 시작 일시, 종료 일시를 포함해 이벤트를 생성할 수 있어야 한다.
- `FR-20` 관리자는 이벤트 결과 화면에서 이벤트별 이벤트명, 시작 일시, 당첨자 수, 참여자 수를 조회할 수 있어야 한다.
- `FR-21` 관리자는 이벤트 결과 화면에서 특정 이벤트의 전체 참여 이력 보기로 이동할 수 있어야 한다.
- `FR-22` 관리자는 특정 이벤트의 당첨자 목록을 조회할 수 있어야 한다.
- `FR-23` 관리자는 전체 참여 이력 화면에서 이벤트명, 시작 일시, 종료 일시, 당첨자 수, 참여자 수를 조회할 수 있어야 한다.
- `FR-24` 관리자는 전체 참여 이력 화면에서 참여자를 선착순으로 정렬된 순위와 당첨 여부까지 확인할 수 있어야 한다.

- `NFR-10` TPS, 응답 시간, 에러율, DB CPU, DB Connections, Slow Query, 락 대기 시간 등의 지표를 수집할 수 있어야 한다.

## 8. 동시성 및 데이터 정합성 원칙

이 프로젝트의 핵심 요구사항은 아래 다섯 가지다.

- `CR-01` 동시에 요청이 들어오더라도 시스템은 최대 당첨 인원을 정확히 지켜야 한다.
- `CR-02` 같은 사용자가 재시도 또는 중복 클릭으로 여러 번 요청해도 한 번만 참여 처리되어야 한다.
- `CR-03` 당첨 가능 수량 확인과 참여 결과 저장은 논리적으로 하나의 원자적 처리처럼 동작해야 한다.
- `CR-04` 처리 도중 실패가 발생하면 부분 반영 상태가 남지 않아야 한다.
- `CR-05` 락 전략, 트랜잭션 경계, 쿼리 구조를 바꾸며 정합성 유지 여부를 검증할 수 있어야 한다.

### 8.1 선착순 판정 기준

선착순은 사용자 단말 시각이 아니라 **서버가 수용한 요청의 DB 반영 순서**를 기준으로 정의한다.

초기 기준은 아래와 같이 둔다.

- 참여 요청이 유효성 검증을 통과한 시점부터 서버 내부 처리 시작
- DB에 기록되는 `request_sequence` 또는 대체 가능한 단조 증가 값으로 정렬 기준 확보
- 당첨 여부 판정은 이 순서와 최대 당첨 인원을 기준으로 결정

이 기준은 구현 후 실험 결과에 따라 보완할 수 있으나, 문서상 공식 기준은 반드시 유지한다.

### 8.2 핵심 설계 원칙

- 중복 참여 방지는 **애플리케이션 로직 + DB 유니크 제약**으로 이중 방어한다.
- 초과 당첨 방지는 **원자적 수량 제어**가 가능한 구조를 우선 검토한다.
- 참여 처리 트랜잭션은 짧게 유지하고 불필요한 조회를 줄인다.
- 락은 정합성을 깨지 않는 범위 안에서 최소 범위로 적용한다.
- 응답 코드와 DB 반영 결과가 불일치하지 않도록 예외 처리 흐름을 명확히 정의한다.

## 9. 예외 및 실패 처리 원칙

- `ER-01` 존재하지 않는 이벤트 요청은 실패 처리한다.
- `ER-02` 인증되지 않았거나 유효하지 않은 사용자 요청은 실패 처리한다.
- `ER-03` 이벤트 시작 전 요청은 명확한 실패 사유와 함께 반환한다.
- `ER-04` 이벤트 종료 후 요청은 명확한 실패 사유와 함께 반환한다.
- `ER-05` 이미 참여한 사용자의 재요청은 중복 참여로 처리한다.
- `ER-06` DB 오류 또는 내부 예외 발생 시 적절한 오류 응답을 반환하고 원인 추적이 가능해야 한다.

- `WIN`
- `LOSE`
- `DUPLICATE`
- `BEFORE_START`
- `ENDED`
- `INVALID_REQUEST`
- `NOT_FOUND`
- `SYSTEM_ERROR`

### 10.2 조회 지원 및 확장 요구사항

- 메인 페이지 조회를 위해 이벤트 상태와 시작 일시 기준 정렬이 가능해야 한다.
- 사용자 이벤트 조회는 별도 상세 화면 없이 목록 중심으로 동작할 수 있어야 한다.
- 관리자 결과 화면을 위해 이벤트별 `winner_count`, `participant_count` 를 빠르게 조회할 수 있어야 한다.
- 전체 참여 이력 화면의 순위 표시는 `event_participations.request_sequence` 기준으로 계산 가능해야 한다.
- 이벤트 대표 이미지는 `events.representative_attach_id` 를 통해 `attach` 를 참조할 수 있어야 한다.
- 당첨 기프티콘 이미지는 `event_participations.gifticon_attach_id` 를 통해 `attach` 를 참조할 수 있어야 한다.
- `attach` 는 별도 파일 서버 없이 백엔드 서버 내부 파일 시스템 경로와 파일 메타데이터를 관리하는 용도로 사용해야 한다.

당첨자 수와 참여자 수 집계는 아래 방식을 비교 실험 대상으로 둔다.

1. `events` 테이블에 현재 당첨 수와 참여자 수 컬럼을 두고 직접 관리
2. `event_participations` 테이블을 기준으로 count 조회
3. 별도 카운터 컬럼 또는 재고성 값을 원자적으로 증가/감소

초기 목표는 가장 단순한 구조로 시작하되, 최종 선택은 **정합성 + 성능 측정 결과**로 결정한다.

### 10.3 DB 설계 원칙

- `users.email` 은 유일해야 한다.
- 동일 이벤트와 동일 사용자 조합은 유일해야 한다.
- 예시 제약: `unique(event_id, user_id)`
- `events.representative_attach_id` 는 `attach.attach_id` 를 참조하는 외래 키로 설계한다.
- `event_participations.gifticon_attach_id` 는 `attach.attach_id` 를 참조하는 외래 키로 설계하되, 미당첨 사용자를 위해 nullable 로 둘 수 있다.
- 이벤트 목록 조회를 위해 `events(status, start_at)` 또는 이에 준하는 인덱스를 검토한다.
- 이벤트별 전체 참여 이력 조회를 위해 `event_participations(event_id, request_sequence)` 인덱스를 검토한다.
- 사용자 마이페이지 조회를 위해 `event_participations(user_id, created_at)` 또는 이에 준하는 인덱스를 검토한다.
- 이벤트별 당첨자 조회를 위해 `event_participations(event_id, result_status, request_sequence)` 인덱스를 검토한다.
- 첨부파일 다운로드를 위해 `attach.attach_id` 기반 단건 조회가 가능해야 하며, 파일 시스템 저장 경로는 `attach.file_path` 로 관리한다.
- 락 범위는 최소화하되 정합성은 반드시 유지한다.
- 트랜잭션은 짧고 단순하게 유지한다.

## 11. API 요구사항 초안

### 11.1 인증 및 회원 API

- `POST /auth/signup` : 회원가입
- `POST /auth/login` : 로그인
- `GET /me` : 내 정보 조회
- `PATCH /me/nickname` : 닉네임 수정
- `PATCH /me/password` : 비밀번호 수정

### 11.2 사용자 이벤트 API

- `GET /events` : 메인 화면 이벤트 목록 조회
- `POST /events/{eventId}/participate` : 이벤트 참여
- `GET /me/participations` : 내 참여 이벤트 및 당첨 결과 목록 조회
- `GET /me/participations/{eventId}/gifticon` : 기프티콘 이미지 조회
- `GET /me/participations/{eventId}/gifticon/download` : 기프티콘 이미지 다운로드

### 11.3 관리자 API

- `POST /admin/events` : 이벤트 생성
- `GET /admin/events` : 이벤트 결과 목록 조회
- `GET /admin/events/{eventId}` : 특정 이벤트 상세 및 집계 조회
- `GET /admin/events/{eventId}/participants` : 전체 참여 이력 조회
- `GET /admin/events/{eventId}/winners` : 당첨자 목록 조회

### 11.4 API 설계 메모

- 인증 방식은 세션 또는 토큰 기반 중 하나로 결정하되, `me` 계열 API는 인증된 사용자 컨텍스트를 전제로 한다.
- `GET /events` 응답은 메인 UI 요구사항에 맞게 진행 중 이벤트 우선, 예정 이벤트 후순위 정렬 또는 구간 분리 응답을 지원해야 하며, 별도 상세 화면 없이 목록만으로 필요한 정보를 제공해야 한다.
- 참여 API는 **멱등하지 않은 요청**으로 취급하되, 동일 사용자 중복 요청에는 동일한 의미의 결과를 반환해야 한다.
- 관리자 참여 이력 API 응답에는 최소 `rank`, `isWinner`, `participatedAt` 정보가 포함되어야 한다.
- 추후 부하 테스트를 위해 응답 본문 또는 헤더에 추적 가능한 요청 ID를 포함하는 방안을 검토한다.

## 18. 포트폴리오 문서화 방향

### 18.1 단순 CRUD와의 차별점

이 프로젝트의 핵심은 짧은 시간 동안 요청이 집중되는 환경에서 **정합성과 성능을 동시에 다룬다**는 점이다.

### 18.2 실험 기반 개선

처음부터 최적 구조를 가정하지 않고, 기본 구조를 만든 뒤 실제 테스트로 병목을 찾고 개선하는 흐름을 보여준다.

### 18.3 기술 선택의 이유

포트폴리오에는 아래 질문에 답이 있어야 한다.

- 왜 이 락 전략을 선택했는가?
- 왜 이 쿼리 구조가 유리했는가?
- 왜 이 인덱스를 추가했는가?
- 왜 트랜잭션 범위를 줄였는가?

### 18.4 수치 중심 결과

결과는 감상이 아니라 수치로 제시한다.

- 평균 응답 시간: A ms -> B ms
- p95: A ms -> B ms
- p99: A ms -> B ms
- TPS: A -> B
- 정합성 오류: 발생 -> 미발생
- 중복 참여 오류: N건 -> 0건

## 21. 마스터 플랜 요약

이 프로젝트는 선착순 이벤트 서비스라는 도메인을 이용해, **동시성 정합성 문제를 실제로 재현하고 개선하는 백엔드 실험형 프로젝트**로 정의한다.  
구현의 목표는 기능 완성이 아니라, **정확한 당첨 처리, 중복 방지, 병목 분석, 개선 실험, 수치 기반 검증**까지 하나의 흐름으로 완성하는 데 있다.
